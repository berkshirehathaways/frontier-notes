# 프론티어노트 콘텐츠 위계 가이드

이 문서는 프론티어노트의 화면 위계와 콘텐츠 타입별 포맷을 정한다. `DESIGN.md`가 색, 타이포그래피, 간격, 컴포넌트 같은 시각 토큰의 기준이라면, 이 문서는 무엇을 먼저 보여주고, 무엇을 작게 두고, 무엇을 줄바꿈하거나 잘라야 하는지를 정하는 편집 디자인 기준이다.

## 1. 핵심 원칙

프론티어노트는 피드가 아니라 편집 아카이브다.

모든 화면은 독자에게 아래 질문을 이 순서로 답해야 한다.

1. 나는 지금 출판물의 어디에 있는가?
2. 내가 보고 있는 대상은 무엇인가?
3. 이 대상은 왜 중요한가?
4. 이 대상은 어떤 이슈, 사람, 도구, 신호와 연결되는가?
5. 다음에 무엇을 읽으면 되는가?

4번의 연결 정보가 2번의 제목보다 커지면 위계가 틀어진 것이다. 메타데이터 때문에 제목 정렬이 무너지면 위계가 틀어진 것이다. 모바일에서 핵심 설명이 잘리고 태그만 남으면 위계가 틀어진 것이다.

## 2. 전체 정보 위계

### Level 0. 전역 정체성

적용 화면: 모든 페이지.

표시 순서:

1. `프론티어노트`
2. `thefutureisnowhere`
3. 주 내비게이션: `ISSUES`, `NOTES`, `PEOPLE`, `ABOUT`, `CONTRIBUTE`

규칙:

- 전역 헤더는 조용해야 한다. 위치를 알려줄 뿐, 콘텐츠와 경쟁하지 않는다.
- 초록 강조색은 브랜드 문장과 상태 배지처럼 드문 곳에만 쓴다.
- 전역 헤더에는 특정 이슈 라벨을 넣지 않는다.

### Level 1. 페이지 헤드

적용 화면: `/notes`, `/people`, `/issues`, `/tools` 같은 아카이브 페이지.

표시 순서:

1. 모노 계열 eyebrow: 페이지 타입과 개수
2. 페이지 제목
3. 짧은 설명
4. 선택적 보조 액션 또는 필터

규칙:

- 아카이브 페이지에서 가장 큰 요소는 언제나 페이지 제목이다.
- 설명은 데스크톱 기준 1~2줄을 넘기지 않는다.
- 필터는 페이지 정체성과 설명 뒤에 온다.

### Level 2. 편집 객체 헤드

적용 화면: 이슈 상세, 글 상세.

표시 순서:

1. 타입 배지
2. 공개 이슈에 포함된 경우 이슈 소속 칩
3. 날짜
4. 제목
5. 부제
6. 증거 이미지 또는 본문

규칙:

- 제목이 주 객체다. 메타데이터가 두 번째 제목처럼 보이면 안 된다.
- 이슈 소속 칩은 타입 배지보다 조용해야 한다.
- `i-am-human`, `issue-01`, `standalone` 같은 내부 값은 공개 UI에 그대로 노출하지 않는다.
- 상세 헤더의 메타데이터는 줄바꿈될 수 있지만, 별도의 헤드라인처럼 보여서는 안 된다.

### Level 3. 목록 행

적용 화면: 최근 노트, 전체 노트, 사람들 목록, 이슈 구성 목록.

표시 순서:

1. 타입 배지 또는 순번
2. 제목
3. 공개 이슈에 포함된 경우 이슈 소속 칩
4. 날짜와 맥락 메타데이터 1개
5. signals, tools, themes는 공간이 남을 때만

규칙:

- 목록 행에서는 제목 정렬이 배지 길이보다 중요하다.
- 타입 배지는 제목 시작점이 흔들리지 않도록 충분히 안정적인 폭을 가진다.
- 이슈 소속은 작은 칩이지 태그 클라우드가 아니다.
- 독립 글에는 가짜 이슈 라벨을 붙이지 않는다.

## 3. 콘텐츠 타입별 포맷

### Issues

목적: 여러 글을 하나의 편집 주장으로 묶는다.

공개 표면:

- `/`
- `/issues`
- `/<issue publicPath>`

히어로 구성 순서:

1. `Issue NN`
2. `Current` 배지
3. 발행일
4. 이슈 제목
5. 커버 이미지
6. 이슈 설명
7. 테마 태그
8. 콘텐츠 구성 목록

데스크톱:

- 공간이 있으면 2열 구조를 쓴다.
- 왼쪽은 제목, 이미지, 설명, 태그를 담는다.
- 오른쪽은 콘텐츠 구성을 담는다.
- 커버 이미지는 장식이 아니라 증거이므로 크게 써도 된다.

모바일:

- 1열로 쌓는다.
- 이슈 설명은 완전히 보여준다. 이슈의 핵심 주장을 자르지 않는다.
- 테마 태그는 한 줄로 유지하고 넘치면 오른쪽에서 자른다.
- 콘텐츠 구성은 제목과 부제를 함께 보여준다.

필수:

- 콘텐츠 구성 순서는 이슈 YAML의 `includedNotes[].order`를 따른다.
- 공개 이슈 라벨은 note frontmatter의 문자열이 아니라 이슈 데이터를 기준으로 만든다.
- 소속 라벨은 `Issue NN · 이슈 제목` 형식을 쓴다.

금지:

- `hidden` 또는 `draft` 이슈를 공개 화면에 보여주지 않는다.
- 예전 placeholder 이슈를 실제 현재 이슈처럼 노출하지 않는다.
- 테마 태그가 제목이나 설명보다 커 보이게 하지 않는다.

### Field Notes

목적: 장면, 관찰, 스크린샷, 현장의 순간을 기록한다.

상세 구성 순서:

1. 타입 배지
2. 이슈 소속
3. 날짜
4. 제목
5. 부제
6. 장면 중심 본문
7. 캡션 있는 증거 이미지
8. 메타데이터 사이드바

목록 구성:

- 타입 배지 다음에 제목이 온다.
- 공개 이슈에 포함된 경우 제목 아래에 이슈 칩을 둔다.
- 대표 맥락 메타데이터는 보통 `location`이다.

모바일:

- 상세 페이지 부제는 자연스럽게 줄바꿈한다.
- 아카이브 목록의 메타데이터는 짧게 유지하고 필요하면 말줄임한다.
- 스크린샷은 증거다. 읽을 수 있는 텍스트를 무리하게 크롭하지 않는다.

필수:

- 장면의 시간 순서를 보존한다.
- 이미지 캡션은 이미지가 왜 증거인지 설명한다.
- 인용문과 설명문은 시각적으로 구분한다.

금지:

- 필드노트를 일반 에세이 카드처럼 만들지 않는다.
- 공개 이슈에 포함된 필드노트의 이슈 소속을 숨기지 않는다.
- 긴 signals를 장면 제목보다 앞세우지 않는다.

### Interviews

목적: 사람의 답변, 목소리, 역할, 이슈와의 관계를 보여준다.

인터뷰는 두 종류로 나눈다.

- 한줄 릴레이 인터뷰: 사람 중심의 짧은 인터뷰.
- 심층 인터뷰: 특정 이슈와 연결되는 긴 대화.

상세 구성 순서:

1. 타입 배지
2. 이슈에 포함된 경우 이슈 소속
3. 날짜
4. 제목
5. 부제
6. 사람 맥락
7. 인터뷰 본문

목록 구성:

- 릴레이 인터뷰 목록은 사람과 제목을 우선한다.
- 심층 인터뷰는 이슈 소속을 더 또렷하게 보여줘도 된다.
- 모바일에서는 공간이 부족하면 일반 이슈 메타보다 `person`과 날짜를 우선한다.

모바일:

- 모든 릴레이 인터뷰에 억지로 이슈 칩을 붙이지 않는다.
- `includedNotes`에 없는 인터뷰에는 이슈 라벨을 만들지 않는다.
- 독립 릴레이 인터뷰 행은 사람 이름과 날짜만으로 충분하다.

필수:

- 사람의 정체성이 읽혀야 한다.
- 릴레이와 심층 인터뷰를 섹션으로 구분한다.
- 사람 아카이브가 필요한 경우 `/people`로 연결한다.

금지:

- 릴레이 인터뷰에서 이슈 라벨이 사람보다 먼저 읽히게 하지 않는다.
- 긴 role/company 문자열을 모바일 첫 줄에 놓지 않는다.

### People

목적: 사람을 아카이브하고 관련 인터뷰와 글로 연결한다.

페이지 구성 순서:

1. 페이지 eyebrow와 개수
2. `사람들` 제목
3. 짧은 아카이브 설명
4. 릴레이 인터뷰와 심층 인터뷰 섹션

목록 구성:

- 인터뷰 카드 포맷을 쓰되 사람 중심 메타데이터를 유지한다.
- 이슈 소속은 해당 인터뷰가 공개 이슈에 포함될 때만 표시한다.

모바일:

- 사람 이름은 긴 signals보다 먼저 보여야 한다.
- 섹션 설명은 짧게 유지한다.

필수:

- People은 Notes의 복제 페이지가 아니라 사람 아카이브여야 한다.
- 릴레이와 심층 인터뷰를 분리한다.

금지:

- 릴레이 인터뷰에서 이슈 소속을 사람 정체성보다 위에 두지 않는다.
- placeholder 인물이나 비공개 연락처를 노출하지 않는다.

### Essays

목적: 프론티어노트의 관점이나 편집 주장을 세운다.

포맷:

- 에세이는 더 큰 cover card를 사용할 수 있다.
- 이슈 소속은 제목이 아니라 메타 영역에 둔다.
- themes는 주장 뒤에 온다.

모바일:

- 제목과 부제를 읽을 수 있게 유지한다.
- cover essay가 이슈에 포함되면 eyebrow 줄에 이슈 칩을 둔다.

필수:

- 글을 열지 않아도 thesis가 보여야 한다.
- 이슈 소속은 에세이가 어느 편집 패키지 안에 있는지 알려주는 용도다.

금지:

- themes 목록이 헤드라인처럼 보이게 하지 않는다.
- 이슈 칩이 이미 있을 때 `issue:` 메타를 반복하지 않는다.

### Systems

목적: 반복 가능한 워크플로, 운영 방식, 기술적 패턴을 설명한다.

포맷:

- 제목과 부제는 사건이 아니라 시스템을 말해야 한다.
- 이슈 소속은 해당 시스템 글이 어느 편집 패키지에 들어가는지 알려준다.
- tools와 signals는 보조 메타데이터다.

모바일:

- 긴 tool 목록은 줄바꿈으로 화면을 밀기보다 잘라낸다.
- 시스템 글은 제목이 추상적일 수 있으므로 부제를 짧고 명확하게 둔다.

필수:

- 시스템 개념이 제목/부제에서 보여야 한다.
- 관찰 장면과 재사용 가능한 원칙을 구분한다.

금지:

- 시스템 글을 리포트처럼 보이게 하지 않는다.
- tool tags를 제목보다 위에 두지 않는다.

### Reports

목적: 여러 글이나 이슈에서 반복되는 패턴을 묶어 종합한다.

포맷:

- 제목은 질문이나 발견을 담는다.
- 이슈에 포함된 경우 이슈 소속을 보여준다.
- themes는 리포트의 핵심 발견 뒤에 온다.

모바일:

- 리포트 행이 태그 중심으로 보이면 안 된다.
- 메타데이터보다 발견이 먼저 읽혀야 한다.

필수:

- 종합의 관점이 드러나야 한다.
- 관련 원문 노트로 돌아갈 수 있어야 한다.

금지:

- 단일 장면 관찰을 리포트로 포장하지 않는다.

### Tools

목적: 도구와 도구가 바꾼 작업 방식을 참고 자료로 정리한다.

페이지 구성 순서:

1. 도구 아카이브 정체성
2. 도구 이름
3. 바뀐 작업 방식
4. 반복 신호
5. 관련 글

규칙:

- Tools는 서사 글이 아니라 레퍼런스 표면이다.
- 표나 compact grid는 스캔 속도를 높일 때만 쓴다.
- 관련 글 카드 안에서 필요한 경우에만 이슈 칩을 보여준다.

## 4. 메타데이터 규칙

### 공개 라벨

공개 UI에서 사용할 수 있는 라벨:

- `Issue NN · 이슈 제목`
- `[FIELD NOTE]`, `[INTERVIEW]`, `[ESSAY]`, `[SYSTEM]`, `[REPORT]`
- `Current`

공개 UI에서 그대로 보여주면 안 되는 값:

- `i-am-human`
- `coaching-the-bot-night`
- `issue-01`
- `standalone`
- 관리자/아카이브 목적이 아닌 collection 폴더명

### 이슈 소속의 기준

공개 이슈 소속은 아래 데이터를 기준으로 판단한다.

```text
src/content/issues/*.yaml -> includedNotes[]
```

노트 frontmatter의 `issue` 필드는 작성과 fallback에는 유용하지만, 공개 UI는 `includedNotes`를 우선해야 한다. 그래야 실제 편집 패키지에 포함된 글만 이슈 라벨을 가진다.

### 독립 글

어떤 published issue의 `includedNotes`에도 없는 글은 다음 규칙을 따른다.

- 이슈 칩을 보여주지 않는다.
- 날짜와 맥락 메타데이터만 보여준다.
- 공개 UI에 `standalone`이라고 쓰지 않는다.

## 5. 모바일 절단 규칙

모바일은 위계가 가장 먼저 무너지는 곳이다. CSS를 바꾸기 전에 아래 표를 기준으로 판단한다.

| 요소 | 모바일 동작 |
|---|---|
| 이슈 제목 | 자연스럽게 줄바꿈 |
| 이슈 설명 | 전부 표시 |
| 이슈 테마 태그 | 한 줄 유지, 오른쪽에서 잘림 |
| 글 제목 | 자연스럽게 줄바꿈 |
| 상세 헤더 부제 | 자연스럽게 줄바꿈 |
| 목록 행 부제 | 필요하면 말줄임 |
| 이슈 칩 | 한 줄 유지, 말줄임 |
| signals/tools | 필요하면 말줄임 또는 숨김 |
| 스크린샷 증거 | 읽을 수 있는 텍스트를 크롭하지 않음 |

모바일 overflow를 해결하기 위해 제목, 부제, 이슈 설명을 숨기지 않는다. 먼저 보조 메타데이터를 줄이거나 자른다.

## 6. 컴포넌트 계약

### `ContentCard`

variants:

- `cover`: 대표 글 또는 아카이브 lead.
- `lead`: 보조 featured 카드.
- `compact`: 홈/아카이브 목록 행.
- `default`: 일반 노트 카드.

필수 동작:

- 이슈 소속은 shared content helper로 계산한다.
- 공개 이슈에 포함된 글만 이슈 칩을 보여준다.
- 이슈 칩과 메타 라인에서 같은 이슈 라벨을 반복하지 않는다.
- `entry-title` 시작점은 타입 배지 길이 때문에 흔들리지 않는다.

### `NoteLayout`

필수 동작:

- 상세 헤더는 raw issue slug가 아니라 공개 이슈 라벨을 쓴다.
- 사이드바의 issue는 보조 참조다.
- 상세 제목은 화면에서 가장 강한 요소로 남아야 한다.

### Issue Feature

필수 동작:

- 현재 이슈 카드는 설명을 전부 보여준다.
- 테마 태그는 시각적으로 보조 요소다.
- 콘텐츠 구성 목록은 제목과 부제를 함께 보여준다.

## 7. 디자인 변경 전 QA 체크리스트

디자인이나 콘텐츠 포맷을 바꾸기 전에 아래를 확인한다.

1. `DESIGN.md`의 토큰과 원칙을 따랐는가?
2. 화면이 먼저 “내가 어디에 있는지”를 알려주는가?
3. 공개 UI에서 raw slug가 숨겨졌는가?
4. 공개 이슈에 포함된 글에는 이슈 소속이 보이는가?
5. 독립 글에는 가짜 이슈 라벨이 없는가?
6. 375px 모바일에서 가로 overflow가 없는가?
7. 제목이 메타데이터보다 강하게 보이는가?
8. 읽을 수 있어야 하는 스크린샷이 크롭되지 않았는가?
9. 같은 콘텐츠 타입이 홈, 아카이브, 상세에서 일관되게 보이는가?
10. `npm run check`와 `npm run build`가 통과했는가?

## 8. 변경 정책

콘텐츠 표면을 바꿀 때는 아래 순서를 따른다.

1. 위계가 바뀌면 이 문서를 업데이트한다.
2. 토큰이나 재사용 컴포넌트 원칙이 바뀌면 `DESIGN.md`를 업데이트한다.
3. 라벨은 페이지별 하드코딩보다 shared helper를 우선한다.
4. 홈, 아카이브, 상세 화면을 함께 확인한다.
5. 이 문서와 충돌하는 일회성 수정은 남기지 않는다.

이 규칙의 목적은 작은 디자인 수정이 다음 디자인 부채가 되는 일을 막는 것이다.
