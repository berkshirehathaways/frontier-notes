# 프론티어노트 콘텐츠 타입 디자인 시스템

이 문서는 프론티어노트의 각 콘텐츠 타입이 어떤 화면 계약을 가져야 하는지 정한다. `DESIGN.md`는 토큰, `content-hierarchy.md`는 전체 정보 위계, 이 문서는 타입별 컴포넌트와 편집 포맷의 기준이다.

## 1. 공통 계약

모든 콘텐츠 타입은 아래 순서를 지킨다.

1. 정체성: 이 글이 어떤 타입인지 알려준다.
2. 제목: 독자가 클릭하거나 계속 읽을 이유를 만든다.
3. 부제: 제목의 장면, 주장, 질문을 좁힌다.
4. 소속: 공개 이슈에 포함된 경우만 이슈 칩으로 연결한다.
5. 맥락: 날짜, 사람, 장소, 도구, signals 중 하나만 먼저 보여준다.
6. 확장 정보: themes, tools, related links는 제목과 부제 뒤에 둔다.

공통 금지:

- 내부 slug를 공개 라벨로 노출하지 않는다.
- signals, themes, tools가 제목보다 먼저 읽히게 하지 않는다.
- 타입 배지 길이 때문에 제목 시작점이 흔들리게 하지 않는다.
- 모바일에서 핵심 설명을 잘라내고 태그만 남기지 않는다.
- 스크린샷을 장식 이미지처럼 크롭해 증거성을 잃게 하지 않는다.

## 2. 타입별 빠른 표

| 타입 | 독자가 먼저 알아야 하는 것 | 카드 우선순위 | 상세 우선순위 | 대표 메타 | 이미지 역할 |
| --- | --- | --- | --- | --- | --- |
| Issues | 이 묶음의 편집 주장 | 이슈 번호, 제목, 설명, 구성 | 제목, 커버, 설명, 구성 | number, current, themes | 이슈의 대표 증거 |
| Field Notes | 어떤 장면이 기록됐는가 | 타입, 제목, 장소/이슈 | 시간순 장면, 이미지, 캡션 | location, issue | 1차 증거 |
| Essays | 어떤 관점을 세우는가 | 제목, thesis, 이슈 | 주장, 근거, 연결 글 | themes, issue | 선택적 분위기 |
| Reports | 어떤 패턴을 종합했는가 | 발견, 범위, 이슈 | 요약, 근거 묶음, 링크 | themes, issue | 선택적 도표/증거 |
| Systems | 어떤 반복 구조를 설명하는가 | 시스템 이름, 짧은 정의 | 구조, 절차, 원칙 | tools, signals | 선택적 다이어그램 |
| Interviews | 누구의 목소리인가 | 사람, 제목, 질문 | 사람 맥락, 질문/답변 | person, interviewKind | 인물/맥락 증거 |
| People | 어떤 사람의 아카이브인가 | 사람 이름, 한 줄 설명 | 프로필, 관련 글 | role, links | 선택적 프로필 |
| Tools | 어떤 도구가 작업을 바꿨는가 | 도구 이름, 용도 | 기능, 사용 맥락, 관련 글 | category, related notes | 선택적 제품 맥락 |

## 3. Issues

목적: 여러 글을 하나의 편집 주장으로 묶는다.

사용 표면:

- 홈 current issue
- `/issues` archive
- 이슈 상세 `/<publicPath>`

카드 계약:

- 메타 줄은 `Issue NN`, `Current`, 발행일 순서다.
- 제목은 카드 안에서 가장 커야 한다.
- 커버 이미지는 제목 바로 뒤 또는 제목과 설명 사이에 둔다.
- 설명은 이슈의 편집 주장을 완전히 보여준다.
- 테마 태그는 보조다. 모바일에서는 한 줄로 유지하고 오른쪽에서 잘라낸다.
- 콘텐츠 구성은 `01 [FIELD NOTE]` 메타 한 줄, 제목, 부제 순서다.

상세 계약:

- 이슈 제목과 설명이 먼저다.
- 콘텐츠 구성은 이슈의 목차이므로 글 카드보다 조용하지만, 제목은 부제보다 선명해야 한다.
- `includedNotes[].order`를 유일한 순서 기준으로 삼는다.

금지:

- placeholder 이슈를 current issue처럼 보이게 하지 않는다.
- 이슈 설명을 말줄임으로 숨기지 않는다.
- 콘텐츠 구성에서 타입 라벨이 제목보다 큰 덩어리로 보이면 안 된다.

## 4. Field Notes

목적: 현장의 한 장면을 시간순으로 기록한다.

사용 표면:

- 최근 노트
- Field Notes archive
- 이슈 콘텐츠 구성
- 상세 글

카드 계약:

- 타입 배지 다음에 제목이 온다.
- 공개 이슈에 포함된 경우 제목 아래에 작은 이슈 칩을 둔다.
- 대표 맥락은 `location` 또는 날짜 하나만 먼저 보여준다.
- signals는 데스크톱에서만 길게 보여도 된다.

상세 계약:

- 제목과 부제 아래에는 장면으로 바로 들어간다.
- 스크린샷은 본문의 증거다. 캡션은 “무엇을 보여주는가”가 아니라 “왜 증거인가”를 말한다.
- 장면 번호를 쓰는 경우 번호는 독서 리듬을 돕는 수준이어야 한다.
- 인용문, 설명문, 편집자 판단은 시각적으로 구분한다.

모바일 계약:

- 부제는 자연스럽게 줄바꿈한다.
- 이미지 캡션은 본문보다 작지만 읽을 수 있어야 한다.
- 긴 signals는 숨기거나 잘라낸다.

금지:

- 필드노트를 일반 에세이처럼 추상적으로 시작하지 않는다.
- 장면 순서를 설명 편의 때문에 뒤섞지 않는다.
- 이미지 하단의 긴 가로선이 소제목 아닌 문단 아래에 남지 않게 한다.

## 5. Essays

목적: 프론티어노트의 관점, 주장, 해석을 세운다.

카드 계약:

- 제목과 thesis가 먼저다.
- 이슈 칩은 있으면 메타 영역에만 둔다.
- themes는 주장의 분류일 뿐 헤드라인이 아니다.

상세 계약:

- 첫 화면에서 “무엇을 주장하는 글인가”가 읽혀야 한다.
- 소제목은 주장 단락의 전환점이어야 한다.
- 이미지가 있다면 증거보다는 맥락 또는 분위기 보조로 쓴다.

모바일 계약:

- 제목은 2줄까지 자연스럽게 허용한다.
- 부제는 제목보다 작지만 잘리지 않는다.

금지:

- 에세이를 리포트처럼 데이터 요약으로 시작하지 않는다.
- themes나 signals를 제목 위의 시각 중심으로 만들지 않는다.

## 6. Reports

목적: 여러 장면이나 글에서 반복되는 패턴을 종합한다.

카드 계약:

- 제목은 발견 또는 질문을 담는다.
- 부제는 종합 범위를 좁힌다.
- 이슈 칩은 관련 패키지로 돌아가는 통로다.

상세 계약:

- 초반에 요약, 근거 묶음, 관련 원문 링크가 보여야 한다.
- 표, 리스트, 도표는 스캔을 돕는 경우에만 쓴다.
- 관련 글로 돌아갈 수 있는 연결을 명확히 둔다.

모바일 계약:

- 리포트 행은 태그 목록이 아니라 발견 중심으로 보여야 한다.
- 긴 themes는 한 줄 이하로 제한한다.

금지:

- 단일 관찰을 리포트로 포장하지 않는다.
- 리포트 제목을 일반 뉴스 제목처럼 뭉뚱그리지 않는다.

## 7. Systems

목적: 반복 가능한 운영 방식, 워크플로, 권한 구조, 기술 패턴을 설명한다.

카드 계약:

- 제목은 시스템 이름처럼 읽혀야 한다.
- 부제는 시스템이 해결하는 긴장이나 반복 구조를 말한다.
- tools와 signals는 보조 메타다.

상세 계약:

- 사건 설명과 시스템 원칙을 분리한다.
- 구조가 복잡하면 다이어그램이나 단계 리스트를 쓸 수 있다.
- 이슈 소속은 해당 시스템 글이 어느 편집 패키지에 들어가는지 알려주는 정도로 둔다.

모바일 계약:

- 부제를 짧고 명확하게 둔다.
- tool 목록은 줄바꿈으로 화면을 밀지 않게 제한한다.

금지:

- 시스템 글을 필드노트처럼 장면만 나열하지 않는다.
- tool tags가 제목보다 먼저 읽히게 하지 않는다.

## 8. Interviews

목적: 사람의 목소리와 작업 방식을 보여준다.

종류:

- Relay interview: 한 사람의 한 문장을 이어가는 짧은 인터뷰.
- Deep interview: 특정 이슈와 연결되는 긴 대화.

카드 계약:

- Relay는 사람 이름과 질문의 선명함을 먼저 둔다.
- Deep은 이슈 소속을 보여줘도 되지만, 사람 이름보다 과하게 커지면 안 된다.
- 제목은 답변의 핵심 장면을 담는다.

상세 계약:

- 사람 맥락은 본문 전에 짧게 둔다.
- 질문과 답변의 위계는 명확해야 한다.
- 편집자 해설은 답변과 시각적으로 구분한다.

모바일 계약:

- 독립 relay 인터뷰에는 억지로 이슈 칩을 만들지 않는다.
- 모바일 카드에서는 `person · date`가 가장 실용적인 메타다.

금지:

- 릴레이 인터뷰에서 이슈 라벨이 사람보다 먼저 읽히게 하지 않는다.
- 답변 없는 인터뷰를 완성 글처럼 노출하지 않는다.

## 9. People

목적: 프론티어노트가 만난 사람과 그 사람의 관련 글을 연결한다.

페이지 계약:

- 페이지 제목은 `사람들`이다.
- 릴레이 인터뷰와 심층 인터뷰를 섹션으로 분리한다.
- 사람 중심 아카이브이지 노트 목록의 복제본이 아니다.

카드 계약:

- 사람 이름, 한 줄 설명, 관련 인터뷰 제목 순서로 읽혀야 한다.
- 이슈 소속은 해당 인터뷰가 공개 이슈에 포함될 때만 표시한다.

모바일 계약:

- 사람 이름과 인터뷰 제목을 우선한다.
- role, company, signals는 과감히 줄인다.

금지:

- placeholder 인물, 비공개 연락처, 내부 메모를 공개하지 않는다.
- 사람 페이지를 태그 목록처럼 만들지 않는다.

## 10. Tools

목적: 도구와 도구가 바꾼 작업 방식을 참고 자료로 정리한다.

페이지 계약:

- 도구 이름이 가장 먼저 읽혀야 한다.
- 도구가 바꾼 작업 방식이 바로 따라와야 한다.
- 관련 글은 도구 설명 뒤에 온다.

카드 계약:

- 제품 소개 카드가 아니라 레퍼런스 카드다.
- category, maker, related notes는 작은 메타로 둔다.

모바일 계약:

- 도구 설명은 한두 줄로 유지한다.
- 관련 글은 너무 많은 칩 대신 compact list로 둔다.

금지:

- 도구 로고나 외부 브랜드가 프론티어노트 정체성보다 커지게 하지 않는다.
- 도구 설명을 광고 문구처럼 쓰지 않는다.

## 11. 화면별 적용 규칙

### Home

- current issue가 가장 큰 편집 객체다.
- 최근 노트는 current issue보다 조용해야 한다.
- 사람들 섹션은 인터뷰의 성격을 분리해 보여준다.

### Archive Pages

- 페이지 제목이 가장 크다.
- 행 제목은 타입 배지보다 중요하다.
- 이슈 소속 칩은 공개 이슈에 포함된 글에만 표시한다.

### Detail Pages

- 상세 헤더는 타입, 이슈 소속, 날짜, 제목, 부제 순서다.
- 본문 안 첫 번째 Markdown 제목이 페이지 제목을 반복하면 제거한다.
- 소제목 아래가 아닌 곳의 긴 가로선은 제거한다.

## 12. 변경 전 체크리스트

새 UI 또는 콘텐츠 포맷을 바꾸기 전에 확인한다.

- 이 변경이 토큰 문제인가, 위계 문제인가, 타입별 포맷 문제인가?
- 제목이 여전히 가장 강하게 보이는가?
- 모바일에서 핵심 설명이 잘리지 않는가?
- 이슈 소속이 공개 이슈 데이터에서 해석되는가?
- 내부 slug나 draft 상태가 노출되지 않는가?
- 스크린샷이 증거로 읽히는가?
- 같은 콘텐츠가 Home, Archive, Detail에서 서로 모순된 위계로 보이지 않는가?
