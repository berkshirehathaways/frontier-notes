# Design Variants: 최전선 노트 리디자인

세 가지 방향을 검토하고, 최전선 노트에 가장 적합한 방향을 선택한다.
**원칙**: Increment와 직접 겹치지 않으면서, 기술 매거진의 효과를 독립적으로 재현한다.

---

## Variant A: Issue Ledger

**개념**: 호 중심 기술 매거진. 홈에서 현재 Issue가 가장 먼저 보이고, 커버스토리 1개와 보조 글 목록이 목차처럼 구성된다.

### 어떤 효과를 노리는가
- 매거진의 편집 단위(호)가 즉각적으로 전달된다.
- "지금 이 호를 읽고 있다"는 감각을 만든다.
- 글 유형보다 호 주제가 전면에 나온다.

### Increment와 직접 겹치지 않기 위해 다르게 설계하는 점
- Increment는 표지 이미지와 색상 배경을 강하게 사용 → 최전선 노트는 이미지를 배제하고 타이포그래피와 구조만으로 표현한다.
- Increment는 "Cover" 컨셉이 일러스트 중심 → 최전선 노트는 코드 에디터 커밋 로그처럼 보이는 발행 기록 + 긴 제목으로 구성한다.
- 목차(TOC)를 인쇄물 목차가 아니라 터미널 `ls -la` 출력처럼 데이터 밀도로 표현한다.

### 장점
- 가장 매거진다운 편집 구조
- 신뢰감이 높다
- 글 수가 적어도 구조가 무너지지 않는다

### 위험
- 호 수가 적을 때 빈 공간이 생길 수 있다
- Increment와 "호 중심"이라는 구조적 유사성이 있다

### 최전선 노트 적합도: ★★★★☆

---

## Variant B: Field Archive

**개념**: 현장 기록과 사람, 도구 데이터베이스가 먼저 보인다. 리서치 아카이브 느낌. People, Tools, Signals가 강하게 드러나는 소싱 엔진.

### 어떤 효과를 노리는가
- "이 사이트는 현장에서 데이터를 모은다"는 인상을 준다.
- 글보다 관찰 데이터가 강조된다.
- 데이터베이스처럼 깊이 있고 지속적인 인상을 준다.

### Increment와 직접 겹치지 않기 위해 다르게 설계하는 점
- Increment는 에세이 중심 → 최전선 노트는 현장 신호 데이터 중심
- DB 항목들을 카드가 아니라 테이블/인덱스 행으로 표현한다 (Airtable-meets-Notion 느낌이 아닌, 학술 인덱스 느낌)
- 날짜가 아니라 "관찰 시점"과 "신호 종류"가 분류 기준이다

### 장점
- 소싱 엔진으로서의 정체성이 가장 강하다
- 글 발행 수가 적어도 People/Tools 데이터로 밀도를 채울 수 있다
- Increment와 가장 다른 방향이다

### 위험
- 매거진보다 데이터베이스 도구처럼 보일 수 있다
- 읽는 경험보다 찾는 경험이 강해진다
- 긴 글 가독성이 상대적으로 약해진다

### 최전선 노트 적합도: ★★★☆☆

## Variant C: Terminal Marginalia

**개념**: 터미널 메타데이터와 현장 노트 감각이 강하다. 젊고 AI native한 느낌. 단, 본문은 매거진처럼 읽혀야 한다. 터미널은 조작 방식이 아니라 시각 언어로만 사용.

### 어떤 효과를 노리는가
- "이 사이트를 만든 사람들도 AI native builder다"라는 신호를 준다.
- 기술 문화 매거진 중 가장 젊고 반-권위적인 느낌이다.
- 도구와 작업 방식을 기록하는 현장감이 시각적으로 전달된다.

### Increment와 직접 겹치지 않기 위해 다르게 설계하는 점
- Increment는 깔끔한 라이트 모드의 에디토리얼 → 최전선 노트는 아이보리 배경 위에 터미널 주석이 여백에 끼어 있는 느낌
- 전체 레이아웃을 터미널로 만들지 않는다 - 메타데이터 라벨, 태그, 신호 표시에만 터미널 감성 사용
- "여백의 주석"(marginalia) 개념: 본문 옆 여백에 코드 댓글처럼 메타 정보가 들어온다

### 장점
- 가장 독특하고 기억에 남는 방향
- AI native builder 독자층과 강하게 공명한다
- 다른 기술 매거진들과 차별화 가능

### 위험
- 과하면 장난감 같아 보인다
- 가독성을 해칠 수 있다
- 일반 독자에게 진입장벽이 생긴다

### 최전선 노트 적합도: ★★☆☆☆ (단독으로는)

---

## 선택: A + B 혼합 + C에서 10% 가져오기

### 선택 근거

**A의 골격** (Issue Ledger 구조): 호 중심 편집 단위, 커버스토리, 목차 감각  
**B의 데이터 레이어** (Field Archive 밀도): People/Tools를 리서치 인덱스로 표현  
**C의 시각 언어 10%**: 메타데이터, 태그, 신호 라벨에만 터미널 느낌 사용

### 비교 기준 점수 (선택된 방향)

| 기준 | 점수 |
|---|---|
| 고급스러움 | ★★★★☆ |
| 정보 밀도 | ★★★★★ |
| 속도감 | ★★★★☆ |
| 신뢰감 | ★★★★★ |
| 현장감 | ★★★★☆ |
| 긴 글 가독성 | ★★★★★ |
| 소싱 엔진 확장성 | ★★★★☆ |

### Increment를 복제하지 않기 위한 구체적 설계 차이

1. **타이포그래피 언어**: Increment는 영문 중심 → 최전선 노트는 한글 세리프가 주 언어, 영문은 보조 라벨
2. **색상**: Increment는 흰 배경에 강한 색상 블록 → 최전선 노트는 아이보리 종이색, 색상 블록 없음
3. **이미지**: Increment는 호마다 커버 일러스트 → 최전선 노트는 이미지 없이 타이포그래피로만 구성
4. **호 표현**: Increment의 표지 → 최전선 노트의 현장 기록 목차 (데이터 밀도로)
5. **세계관 문장**: `thefutureisnowhere` - Increment에는 없는 독자적 정체성 요소
6. **메타데이터 스타일**: 터미널 라벨 감각 (tools: , signals: , issue: ) - 최전선 노트만의 시각 언어
7. **People/Tools 인덱스**: 잡지 인물 소개 방식이 아닌 데이터베이스 행 형식

---

## 구현 우선순위

1. **홈 화면**: 호 중심 편집 구조로 재편
2. **글 상세 페이지**: 긴 글 가독성 최우선 (prose 타이포 수정)
3. **ContentCard**: 편집 위계 (cover / lead / compact)
4. **Issues 페이지**: TOC 스타일
5. **BaseLayout 헤더**: 매거진 masthead + world statement
6. **Research Index**: People/Tools를 데이터 인덱스로
7. **모바일**: 모든 핵심 레이아웃 모바일 대응
