# Design Audit: 최전선 노트 리디자인

> **원칙 선언 — 카피 금지**
> 이 리디자인은 특정 사이트를 복제하지 않는다. Increment에서 참고하는 것은 레이아웃 원리, 시각 언어, 정보 밀도, 호 중심 편집 구조, 긴 글 가독성, 기술 매거진의 인상이다. 모든 UI는 최전선 노트의 정체성에 맞게 독립적으로 재해석한다.

---

## 1. 현재 사이트가 기술 매거진처럼 보이지 않는 이유

1. **Hero = SaaS 랜딩페이지 문법**: 홈 상단이 "큰 제목 + 부제 + CTA 버튼 2개" 구조로 이루어져 있다. 이는 제품 소개 페이지의 전형적인 패턴이다. 매거진 첫 화면에 CTA 버튼이 정면에 자리 잡으면, 독자는 "읽는 공간"이 아닌 "구매/가입하는 공간"으로 인식한다.
2. **Signal Intake 블록 = 제품 기능 그리드**: "People DB / Tools DB / Field Signals / Newsletter Archive"를 2×2 그리드로 배치한 것은 SaaS 제품 Feature 섹션과 동일한 시각 언어다. 매거진은 편집 섹션을 이런 방식으로 소개하지 않는다.
3. **호(Issue)가 시각적으로 묻혀 있다**: 현재 구조에서 Current Issue는 홈 Hero의 오른쪽 사이드 패널로 축소되어 있다. 기술 매거진에서 호는 편집 단위의 핵심이어야 한다.
4. **균일한 3열 카드 그리드 반복**: Interviews, Field Notes, Systems, People, Tools가 모두 같은 크기의 카드 그리드로 나열된다. 편집 위계가 없으면 무엇이 중요한지 독자가 알 수 없다.
5. **내비게이션이 너무 크고 기능적이다**: primary-nav가 link-bar 스타일로 기능하지만, 시각적 무게감이 매거진의 얇고 절제된 헤더가 아닌 일반 웹사이트 헤더처럼 보인다.

---

## 2. 현재 사이트가 블로그 / SaaS / 포트폴리오처럼 보이는 지점

| 문제 지점 | 어떻게 보이는가 |
|---|---|
| 홈 Hero frame + CTA 2개 | SaaS 랜딩페이지 |
| Signal Intake 2×2 그리드 | 제품 기능 섹션 |
| `btn` 클래스의 채워진 검은 버튼 | 제품 CTA |
| 모든 섹션이 동일한 3열 카드 | 개발자 블로그 |
| 뉴스레터 폼이 카드 안에 내장 | SaaS 플랜 등록 섹션 |
| 전체 레이아웃이 "구역 단위" | 에이전시 포트폴리오 |

---

## 3. 현재 정보 구조의 문제

- **호(Issue) 중심성 부재**: 기술 매거진은 호가 편집 단위다. 현재는 글 유형(essays, interviews 등)이 기준이고, 호는 부속 정보처럼 보인다.
- **Featured/Lead 글이 다른 글과 동일한 크기**: 편집적 위계가 없어 모든 글이 같은 중요도처럼 보인다.
- **People / Tools가 글과 같은 깊이에 배치**: 리서치 인덱스 성격의 People, Tools DB는 글 섹션과 같은 방식으로 카드화되어 있어, 아카이브다운 밀도감이 없다.
- **thefutureisnowhere 세계관이 어디에도 없다**: 이 사이트의 핵심 정체성 문장이 현재 완전히 부재하다.

---

## 4. 현재 홈 화면의 문제

- 첫 화면이 "이 사이트가 무엇을 하는 매거진인가"보다 "지금 구독/읽기를 클릭하라"고 말하고 있다.
- 현재 Issue가 홈에서 가장 눈에 띄어야 하는데, 오른쪽 사이드 패널로 밀려 있다.
- 커버스토리(Featured Essay)가 일반 ContentCard와 같은 크기로 렌더링된다.
- 세계관 문장(`thefutureisnowhere`)이 없다.
- 내비게이션 위에 `top-alert`(NEW 뱃지)가 있는데, 이것이 쇼핑몰 알림처럼 느껴진다.

---

## 5. 현재 글 상세 페이지의 문제

- `detail-head` 영역의 제목이 너무 작다 (max `2.95rem`).
- 본문(prose) 내 `h2, h3`가 **모노스페이스 폰트**로 렌더링된다 - 매거진 기사 느낌을 크게 훼손한다.
- 본문 최대 너비(`max-width: 72ch`)는 적절하나, 패딩과 행간이 더 필요하다.
- 사이드바(metadata, 다음 질문, CTA)가 본문과 같은 시각 무게로 경쟁한다.
- 메타데이터 패널이 `.frame` 스타일로 박스화되어 있어 딱딱하다.

---

## 6. 현재 카드 디자인의 문제

- 모든 카드가 동일한 크기와 구조 → 편집적 위계 없음
- `note-meta` 박스가 배경색이 있어 너무 무겁다 (배경: `--tag`)
- 카드 하단 태그 목록이 본문 제목과 시각적으로 경쟁한다
- 카드 패딩이 균일해서 "정보 밀도"가 없다
- 카드 자체가 정보를 충분히 보여주지 않는다 (제목 + 부제 + 메타 외에 내용 프리뷰 없음)

---

## 7. 현재 타이포그래피의 문제

- **본문 헤딩(prose h2, h3)이 모노스페이스**: `font-family: ui-monospace`로 설정되어 있음 - 기술 매거진 기사에서 소제목이 모노스페이스면 코드 블록처럼 보인다.
- **행간이 충분하지 않다**: `prose`의 `line-height: 1.86`은 괜찮지만 `font-size: 1.02rem`과 결합했을 때 한글 본문에서는 조금 타이트하다.
- **본문 폰트 크기**: 글 상세 페이지에서 `1.02rem`보다 `1.06rem` 정도가 한글 가독성에 더 좋다.
- **메타데이터 폰트**: 현재 `mono` 클래스 사용은 좋은 방향이나, 크기(`0.69rem`)가 너무 작아 읽기 어렵다.
- **섹션 제목 행간**: `section-title h2`의 `line-height: 1.12`는 한글에 너무 타이트하다.

---

## 8. 이번 리디자인에서 유지할 기능

- 모든 라우팅 구조 (essays, interviews, field-notes, systems, reports, issues, people, tools, newsletter)
- Keystatic CMS 연동 구조
- content collections 타입 정의
- `getPublishedCollection`, `resolveIssueNotes` 등 콘텐츠 유틸
- 기본 색상 팔레트 방향 (아이보리 배경, 세리프 제목, 모노 메타)
- `--radius: 0` (각진 모서리 유지)
- `--shadow: none` (그림자 없음 유지)
- 모바일 반응형 breakpoints
- 접근성 (focus-visible 스타일, aria 레이블)
- 무료 운영 구조 (외부 DB, 로그인, 결제 없음)

---

## 9. 이번 리디자인에서 건드리지 않을 기능

- Keystatic CMS 설정 파일 (keystatic.config.ts)
- content.config.ts (콜렉션 스키마)
- 모든 MDX 콘텐츠 파일
- lib/content.ts, lib/site.ts 로직
- astro.config.mjs
- 인터뷰 제안 폼 (InterviewProposalForm.astro) 동작 로직
- 뉴스레터 폼 동작 로직 (UI만 정리)
- 개별 글 페이지 라우팅 ([slug].astro 패턴)
- notes 탐색기 (NotesFilter.astro)
